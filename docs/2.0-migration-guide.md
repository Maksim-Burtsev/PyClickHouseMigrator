# 2.0 Migration Guide

This guide explains the current SQL-first direction of PyClickHouseMigrator and what that means for contributors and reviewers.

## Product Direction

The current product story is:

- lightweight
- ClickHouse-native
- SQL-first
- no ORM
- no schema diff engine

The migrator is focused on applying and tracking migrations, not on replacing GUI database tools or general-purpose schema export workflows.

## Current Migration Contract

Migrations are `.sql` files with two named sections:

```sql
-- migrator:up
-- @stmt
CREATE TABLE ...

-- migrator:down
-- @stmt
DROP TABLE ...
```

Rules:

- exactly one `-- migrator:up`
- exactly one `-- migrator:down`
- `up` must contain at least one non-empty `-- @stmt` block
- every non-empty statement must start with `-- @stmt`
- `down` may be empty

Pending migrations are applied in lexicographic filename order.

Applied history, `HEAD`, and rollback order are determined by stored apply timestamps in `db_migrations`.

## What Changed in the SQL-First Direction

The important changes in the current direction are:

- `.sql` is the primary migration format
- marker-based statements replaced blind SQL splitting
- checksum validation is based on parsed statement lists
- `baseline` exists for onboarding an already-existing schema
- `show` exposes `HEAD`, `baseline`, and integrity status

## Baseline Semantics

`baseline` is for existing databases.

It:

- requires an empty `db_migrations` table
- writes ledger rows only
- does not execute migration SQL
- allows future `up` runs to apply only newer migrations

It is intentionally different from `--dry-run`:

- `--dry-run` prints and validates SQL, but writes nothing
- `baseline` writes ledger state, but executes no SQL

## What Is Intentionally Out of Scope

PyClickHouseMigrator does not currently try to be:

- a GUI schema browser
- a general-purpose schema export tool
- an ORM migration layer
- a schema diff engine
- a deployment orchestrator

That boundary is deliberate. The tool should stay small and predictable around migration execution and migration state.

## Current Release-Surface Priorities

For the current phase, the most useful documentation surface is:

- clear SQL-first README
- deployment recipes
- contributor workflow
- release notes and migration guide

Less useful work should stay de-prioritized unless a real need appears.
